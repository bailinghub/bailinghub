# 百灵中枢 · 第三方对接指南

> 这份文档是给**第一次对接百灵中枢的业务开发者**看的。你不需要先读懂中枢的内部设计——按本文从头走一遍即可把你的业务接进来。
> 遇到字段级的精确定义，再去查 `CONTRACT.md`（边界契约，唯一权威）。本文负责"去哪点、怎么接、怎么验、怎么排错"。

---

## 0. 开始之前：准备这两样东西

| 需要准备 | 用途 |
|---|---|
| **中枢地址**（形如 `https://hub.example.com`，下文统一写 `<中枢地址>`） | 后台登录 + 业务系统调用的根地址 |
| **一个后台账号**（用户名 + 初始密码） | 登录 `<中枢地址>/console/` 做配置 |

> 全文凡是 `<中枢地址>` 都替换成实际部署地址。后台账号登录后请先修改初始密码。

**对接前先想清楚三件事**（决定你走哪条路，见 §2）：
1. 你是想让**业务后端**把一个事件丢给 AI 处理、拿回结果？（→ 形态 A）
2. 还是想在**网页 / H5 / 公众号**里放一个聊天框给用户用？（→ 形态 B）
3. 还是想让 **AI 能直接查 / 操作你的业务数据**（查订单、改配置…）？（→ 形态 C，需要你这边写少量后端代码）

三者可以叠加，但建议先把一条跑通。

---

## 1. 五个核心概念（读这一段就够开始了）

```
你的业务 ──触发──▶  中枢（按路由派给某个 AI 大脑去干）──▶  结果回流给你
                         │
                         └─ 大脑可以是中枢内置的云端 LLM，也可以是外部执行器或本地智能体
```

| 概念 | 一句话 | 在后台哪里 |
|---|---|---|
| **接入方（Client）** | 你的业务系统的一把"钥匙"（token），限定能调哪些路由、限速 | 「接入方」页 |
| **触发路由（Route）** | 一个业务场景：发给哪个大脑、用不用知识库、结果送哪、会话怎么续 | 「触发路由」页 |
| **调度目标（Target）** | 一个"大脑插孔"。出厂自带一个 `llm`（中枢内置云端 LLM），够用就不用动 | 「调度目标」页 |
| **工具源（Tool Provider）** | 你按 ACC 声明给 Agent 的“可调接口清单”，让 Agent 从“会答”变成“能查、能办”（形态 C 才用） | 「工具源」页 |
| **对话总账** | 所有往来自动落库，AI 会话只是缓存——可重放、可追溯，你不用自己存上下文 | 「任务」页可查 |

**最重要的一句**：你的业务和中枢**互不引用代码**，只通过 HTTP 说话；唯一的"缝"是冻结的信封契约（`CONTRACT.md`）。中枢挂了你的业务也不能崩——所有调用都是**单向、异步、可降级**的。

## 2. 选择你的对接形态

| 你的需求 | 形态 | 你要做的活 | 看本文哪节 |
|---|---|---|---|
| 业务后端：某事件发生 → 让 Agent 处理 → 拿回结构化结果 | **A · 异步 Agent 工人** | 后端发一个 HTTP POST + 接收回调/轮询 | §4 |
| 网页/H5/小程序里放个聊天框 | **B · 网页聊天组件** | 贴一行 `<script>`（要带登录身份再加一点后端） | §5 |
| 让 Agent 能查/办你的业务（查订单、改配置、发券…） | **C · 工具源** | 给接口加 ACC 声明 + 后端实现“验签+授权” + 后台注册 | §6 |
| 业务后端：主动把一条消息经渠道（企微等）推给某用户，且让它进会话历史 | **D · 主动出站** | 后端发一个 HTTP POST（渠道 + 收件人 + 正文） | §7 |

先跑通一条，再叠加。下面每节都是**后台配置 + 业务代码 + 联调**完整闭环。

## 3. 第一步（所有形态都要）：登录后台 + 确认模型可用

1. 浏览器打开 `<中枢地址>/console/`，使用中枢管理员创建的账号登录。
2. 看「**模型凭证**」页：
   - 如果里面**已经有可用凭证** → 跳过这步。
   - 如果是空的 → 点新增，填一个 **OpenAI 兼容**的端点 + key（OpenAI / DashScope通义 / OpenRouter / 你自建的都行）。**配完这一项，内置的 `llm` 大脑就能用了，不需要任何额外部署。**

> 之后所有形态都建立在"路由有一个能用的大脑"之上。默认就用 `llm`，等你有本地智能体要接再说（§8）。

## 4. 形态 A：业务后端要个"异步 AI 工人"

**场景举例**：来了个工单 → 丢给 AI 做初判 → 拿回 `{严重程度, 根因假设, 建议}` → 你的系统据此自动归类或升级人工。

### 4.1 后台配置（点几下）

1. 「**触发路由**」→ 新建：
   - **路由 key**：起个名，如 `ticket-triage`（这是你调用时要带的 `route`）。
   - **调度目标**：选 `llm`。
   - **会话策略**：一问一答选 `new`；要按业务键多轮续聊选 `per_key`（见 §9）。
   - （可选）挂**知识库**、配**结果送达**、配**重试**——先不配也能跑。
2. 「**接入方**」→ 新建：
   - 起名（如 `my-shop`），**勾选刚建的路由**进白名单，设每分钟限速。
   - 保存后得到一个 **token**（形如 `bz_xxx...`）——这就是你后端要用的钥匙，**只放服务端，别进前端**。
3. 回到「触发路由」那一行，点「**调用代码**」按钮——它会生成可复制的 curl / SDK 调用示例（已内置超时和异常处理建议）。下面给等价示例帮你理解字段。

### 4.2 业务后端调用：`POST /run`

```bash
curl -X POST "<中枢地址>/run" \
  -H "Authorization: Bearer <你的接入方token>" \
  -H "Content-Type: application/json" \
  -d '{
    "request_id": "myshop-ticket-10086",     // 幂等键！同 request_id 重复调 = 返回同一个 job，不重复跑
    "route": "ticket-triage",                  // 你建的路由 key；也可传 "auto" 让中枢按受众/关键词分诊
    "input": "用户反馈：会员卡退款失败，提示 token 过期",   // 业务正文，会被当作不可信数据，不作为指令
    "metadata": {
      "ticket_id": "10086",
      "principal": { "id": "u-1001", "tenant": "t-179", "roles": ["cs"], "audience": "employee" }
    },
    "callback_url": "https://yourapp.com/bailing/callback"     // 可选：完成后把结果 POST 给你；不填就轮询
  }'
```

**立即返回（HTTP 202），绝不同步等结果**：

```json
{ "job_id": "uuid", "status": "queued", "request_id": "myshop-ticket-10086", "route": "ticket-triage", "target": "llm", "session_id": "uuid", "continue": false }
```

关键纪律：
- **`request_id` 必填且要全局唯一**（建议加你自己的 app 前缀防撞号）。重复 = 幂等返回同一个 job。
- **接入方只能带 `route`**，不能自带 `project`/`profile`（那是后台说了算）→ 违者 403。
- **主体建议统一放 `metadata.principal`**：`id` 是业务用户/操作员稳定 ID，`tenant` 是租户/组织，`roles` 是角色数组，`audience` 是员工/C端/管理员等受众类型。路由的受众策略和自动分诊都按这个标准形状判断。
- **`route:"auto"` 不是让 Agent 随便选**：它只在你这个接入方白名单内、且路由显式配置了受众 `auto/keywords` 的候选里按规则分诊；同分会返回 409，要求后台补优先级或关键词。
- 限速超了 → **429**，稍后用**同 `request_id`** 重试即可（不会重复跑）。
- 中枢被管理员暂停（kill switch）→ **503 `{"status":"paused"}`**，你应**降级到人工队列，别重试风暴**。

### 4.3 拿结果：回调 或 轮询（二选一）

**方式①：回调（推荐）** — 你在 `/run` 里填了 `callback_url`，完成后中枢把**完整 job 对象** POST 给你，并带签名头。你**必须验签**：

```php
<?php
// 中枢回调/送达的签名：标签 sha256=；时间戳是【毫秒】，材料 = "<毫秒时间戳>.<原始请求体>"
$ts   = $_SERVER['HTTP_X_BAILING_TIMESTAMP'] ?? '';
$sig  = $_SERVER['HTTP_X_BAILING_SIGNATURE'] ?? '';
$body = file_get_contents('php://input');
$myToken = '<你的接入方token>';   // 用你调 /run 那把 token 验签

$mac = hash_hmac('sha256', $ts . '.' . $body, $myToken);
if (!hash_equals('sha256=' . $mac, $sig) || abs(time()*1000 - (int)$ts) > 300000) {
    http_response_code(401); exit;   // 验签失败或时间戳超 300 秒，拒绝
}
$job = json_decode($body, true);
// $job['status'] === 'done' → 读 $job['report']（结构化结论）；'error' → 看 $job['error']
// 按 $job['metadata']['ticket_id'] 找回你自己的业务记录
http_response_code(200);
```

**方式②：轮询** — 不填 `callback_url`，过几秒后 `GET <中枢地址>/jobs/<job_id>`（带你的 token），看 `status` 到 `done`/`error` 再读 `report`。

### 4.4 结果长什么样

`status=done` 时 `job.report` 是结构化结论（详见 `CONTRACT.md §4`）：

```jsonc
{
  "summary": "会员卡退款失败疑似 token 过期，需刷新鉴权",
  "severity": "P2",
  "category": "配置",
  "root_cause_hypothesis": "…",
  "confidence": 0.82,
  "needs_human": false,        // 解析失败/拿不准时强制 true → 你应升级人工
  "suggested_next_step": "…"
}
```

你的系统据此自动路由：高 `confidence` + 低 `severity` 自动归类；`needs_human=true` 转人工。

## 5. 形态 B：网页 / H5 放一个聊天框

**场景举例**：官网客服悬浮窗、H5 帮助问答。访客直接打字，AI 同步回答。

### 5.1 后台配置

1. 先有一条路由（同 §4.1 第 1 步，目标 `llm`；要让 Agent 答得准就给路由挂个**知识库**）。
2. 「**聊天入口**」→ 新建，绑这条路由：
   - 得到一个**公开的 `entry_key`**（形如 `pub_xxx`，**可公开、无需 token**）。
   - 配**站点 Origin 白名单**（防别的网站盗用你的入口）、**IP 限速**。
   - 可选配**外观**（窗口尺寸/位置/头像/标题）、**开场白**、**主色**。

### 5.2 嵌入网页（一行）

```html
<script src="<中枢地址>/widget.js" data-entry="pub_xxx" async></script>
```

贴进任何网页即出现聊天组件。访客默认是**匿名身份**，自动按浏览器续聊。**到这里就能用了。**

### 5.3 （可选）让登录用户带可信身份进聊天

如果你想让聊天里的用户是"已登录的张三"（以便日后接工具时知道是谁在操作），需要你后端**签一张短票**：

1. 后台聊天入口里配一个「**票据签发方**」（指向你的某个接入方 app_id）。
2. 你的网页在**登录态的后端**用**自己的接入方 token** 签票：

```php
// payload = base64url( {"uid":"179:1001","exp":<unix秒过期>} )   // 多租户建议用 "租户:用户" 结构化主体，见下
$payload = rtrim(strtr(base64_encode(json_encode([
    'uid' => '179:1001',                 // 你的操作人标识，中枢原样回传，格式你定
    'exp' => time() + 3600,
])), '+/', '-_'), '=');
$ticket = 'v1.' . $payload . '.' . hash_hmac('sha256', $payload, '<你的接入方token>');
```

3. 网页把票据输出给组件：

```html
<script src="<中枢地址>/widget.js" data-entry="pub_xxx" data-ticket="<上面签出的票据>" async></script>
```

验签通过后，这个 `uid` 会成为后续工具调用的"操作主体"，会话也按人续（换设备也接得上）。**接入方 token 永远不要进前端**——只在后端签票。

> **多租户务必用"结构化主体"**：别只签裸 `uid`，签 `"{tenant}:{user}"`（如 `"tenant_179:user_1001"`）。否则接第二个租户时 `uid=1` 会跨租户撞车。详见 `CONTRACT.md §2.4`。

## 6. 形态 C：让 Agent 能查 / 办你的业务（工具源）

> **先说实话**：这一步不是“在后台点点点”就完事的。后台只负责**注册和配额**；让 Agent 安全地调你的接口，**业务侧要写后端代码**：给接口加 ACC 声明 + 实现“**验签 + 授权**”两道闸 + 发布接口清单。这是个小工程，不是粘一段中间件。官方 SDK 会把最容易错的部分（签名、授权）钉死。

### 6.0 职责劈分（记住这一句，能省一半的坑）

- **中枢管"这路 AI 最多够到哪"**：白名单 / 风险闸 / 限流 / 审计 / 给你的请求签名。
- **你管"这个人此刻能不能做这件事"**：验签确认"真是中枢发来的"之后，**用你自己的权限表裁决**。
- ⚠️ **致命陷阱**：工具调用是**服务器到服务器、没有登录 session**！你平时基于 Cookie/session 的鉴权对这条链路**完全不生效**。所以"把操作主体接进你的权限表"是这条链路**唯一**的授权闸——**只验签不授权 = 认证了但没授权**，等于裸奔。

### 6.1 给接口加注解（声明哪些接口给 AI、什么权限、什么风险）

在你的 OpenAPI spec 的每个 operation 上加 `x-agent-capability`（用我们的 PHP SDK 则是 `#[AiTool(...)]` 注解，自动生成）：

```yaml
# 一个"查员工列表"的只读接口示例
/staff/list:
  get:
    operationId: staff_list          # 工具名，定了就别改（改名会让 Agent 认为这是另一个新工具）
    summary: 查询门店员工列表          # AI 判断何时调用的首要依据，写人话
    x-agent-capability:
      version: 1
      enabled: true                 # 必填：不标 = 永远不暴露给 AI
      scope: tenant.staff.read      # 必填：权限标签，路由 allow 白名单按它匹配
      risk:
        level: low                  # low(默认) / medium(放行留痕) / high(进入高风险审批承接，见 §6.2.2)
      execution:
        readonly: true              # 只读声明，Agent 可放心调
    parameters:
      - { name: dept, in: query, schema: { type: string }, description: 部门名 }
```

完整注解注册表（含写接口、审批提示、敏感数据、超时等）见 `CONTRACT.md §2.4a`。要点：
- **写操作要审批**？见 §6.2.2。接口本身是“提交申请”时，由业务接口内部创建审批单并返回待审核；接口是真实执行动作时，可用 `risk.level: high` / `approval.required` / `approval.when` 让中枢先冻结调用意图，再投递给业务侧审批流。
- 涉及操作人的接口在 ACC 里标 `subject.required: true`，匿名访客就看不到它。
- 非 GET 接口**至少要声明一个参数**，否则不暴露（不让 Agent 瞎猜）。
- 不建议把底层 CRUD 原样暴露给 AI。优先暴露“查询 / 预检 / 申请 / 草稿”这类业务门面；真正执行型接口再考虑 `high` 或 `confirm-required`。

你不需要一开始填满所有字段。最小可用路径是：选一个后台已有动作 → 在 ACC 里标 `enabled` / `scope`，再写好 `summary` / 参数 schema → 在接口里验签并按 `X-Bailing-On-Behalf-Of` 走原有权限表。AI 本质上是在替这个操作主体点同一套后台按钮；能不能操作仍由你的业务系统判断。

#### 6.1.1 先设计 Agent 工具，再贴 ACC 字段

同一个业务能力可以有几种接口形态，推荐从低风险形态开始：

| 工具形态 | 示例 | 推荐声明 |
|---|---|---|
| 查询 | `order.get`、`staff.search` | `execution.readonly:true`，`risk.level: low` |
| 预检/试算 | `refund.preview`、`staff_remove.impact`、`price_change.estimate` | `low` 或 `medium` |
| 申请/草稿 | `refund.request.create`、`staff_remove.request` | `medium`，业务侧审批流承接 |
| 真实执行 | `refund.execute`、`staff.delete`、`permission.grant` | 通常 `high` 或 `approval.required:true` |
| 批量执行 | `coupon.batch_send`、`price.batch_update` | 通常 `high`，并配 `approval.when` |

业务接口如果只是创建申请，响应要明确告诉 AI “待审批”，不要让它误以为已经执行完成：

```json
{
  "ok": true,
  "status": "pending_approval",
  "message": "已提交退款申请，等待主管审核。",
  "business_id": "refund_req_1001"
}
```

批量接口要特别保守：不要让 Agent 传裸 SQL 或任意过滤表达式；用明确参数、枚举、数量上限和预检接口表达影响范围。

### 6.2 实现"验签 + 授权"中间件（核心，用 SDK）

PHP 业务直接用官方 SDK（`sdk/php/`，零依赖；PHP7 用 `sdk/php7/`）。**一个方法搞定验签 + 授权**：

```php
<?php
use Bailing\Verify;

// 你的工具接口入口（如 ThinkPHP 中间件 / 控制器前置）
$secret = '<工具源签名密钥>';   // 后台注册工具源时你设的，和接入方token、server.token 互相独立

$result = Verify::gate(
    $secret,
    // ⚠️ authorize 回调是【必填】的授权闸。切勿写成 fn() => true！
    function (string $operator, string $tool, array $params): bool {
        // $operator = X-Bailing-On-Behalf-Of，即"替谁操作"。多租户先拆租户再认人：
        // 例：$operator = "179:1001" → 租户179 的用户1001
        [$tenant, $uid] = array_pad(explode(':', $operator, 2), 2, '');
        if ($tenant === '' || $uid === '') return false;          // 空主体直接拒（按你的语义）
        return MyPermission::can($tenant, $uid, $tool, $params);   // 用你既有的权限表裁决
    },
    $toolNameForThisEndpoint     // 当前端点对应的工具名（operationId）
);

if (!$result['ok']) {
    // $result['code'] = 401(验签失败) 或 403(授权拒绝)；$result['error'] 是原因
    http_response_code($result['code']);
    echo json_encode(['error' => $result['error']]);
    exit;
}
// 走到这里：既"真是中枢发的"，又"这个主体有权做" → 执行你的真实业务逻辑
```

非 PHP 语言照 `CONTRACT.md §2.4b` 实现验签——`docs/examples/` 下有 **Node（`.mjs`）、Python（`.py`）、PHP（`.php`）三份零依赖单文件参考**，每份直接运行即比对契约里的冻结测试向量，照着翻 ~40 行即可，无需读 PHP。

> **签名**：中枢调用你的接口带 **`sha256=` 签名**（算法名、全框架统一、非版本号；构造把操作主体和任务号也钉进 HMAC，防换租户/重放）。SDK `Verify::gate()` 与三份单文件参考都按这一套实现——你照着实现 `sha256=` 即可。

### 6.2.1 操作主体的命名空间（网页票据 vs 企微等渠道入站）

**同一个工具，可能从不同入口被触发；`X-Bailing-On-Behalf-Of` 的主体会带不同命名空间，你的 `authorize` 必须都认。**

| 触发入口 | 你收到的 `$operator` 形如 | 含义 | 你要做的解析 |
|---|---|---|---|
| 网页聊天 / 你自己 /run 调用（有登录票据） | `179:1001`、`pa:u2`（**你票据里声明的**） | 已是**业务命名空间**的身份 | 直接拆租户/uid → 查权限 |
| 企微入站（**无登录票据**） | `wecom:<企微userid>`（如 `wecom:user_001`） | 中枢从**企微官方签名回调**派生的可信身份，但在**企微命名空间** | **先映射到业务 uid**，再查权限 |

为什么会这样：企微用户没有登录、没有票据，但企微官方回调的签名保证了"这条消息确实来自该 userid"。中枢据此派生 `wecom:<userid>` 作为操作主体透传给你——**它是可信身份，只是命名空间是企微的，需要你这边翻译成业务账号**（通常查你企微通讯录同步落下的 `wx_userid ↔ 业务uid` 映射）。网页路径省了这一跳（票据直接带了业务 uid），企微路径多这一跳，之后**走你同一张权限表**。

```php
function (string $operator, string $tool, array $params): bool {
    if ($operator === '') return false;                       // 空主体直接拒
    // 企微入站：wecom:<userid> → 先翻译成业务 uid，再走和网页同一套鉴权
    if (str_starts_with($operator, 'wecom:')) {
        $wxUserid = substr($operator, strlen('wecom:'));
        $uid = MyBinding::businessUidByWecom($wxUserid);      // 查 wx_userid↔业务账号 映射（通讯录同步那张表）
        if (!$uid) return false;                              // 未绑定 → 拒（或走 step-up 让用户先认证绑定）
        return MyPermission::canByUid($uid, $tool, $params);
    }
    // 网页票据：你自己定义的命名空间（如 租户:uid）
    [$tenant, $uid] = array_pad(explode(':', $operator, 2), 2, '');
    if ($tenant === '' || $uid === '') return false;
    return MyPermission::can($tenant, $uid, $tool, $params);
}
```

注意：
- **安全**：`wecom:` 主体只会出现在**企微官方来源**的调用里——中枢用任务 `source` 把关（`/run` 客户端即便在 metadata 里塞 `wecom_userid` 也不会被采信），所以你拿到 `wecom:X` 即可信其来自企微签名回调；但你这边仍要 **fail-closed**（映射不到就拒，绝不 `return true`）。
- **共享路由**：若一条路由同时被网页聊天和企微用（很常见），上面两个分支都要在；缺任一分支那条入口就会被你误拒或漏判。
- **未绑定的人**：`businessUidByWecom` 查不到 → 返回 false（拒）。要更友好可让大脑回一个企微 OAuth 深链做**一次性身份绑定**（step-up），认证后落下映射，之后即走上面的快路径。
- **高危写操作**（如开户）：接口同时标 `x-agent-capability.subject.required: true`；中枢对**无主体**的任务根本不暴露该工具（fail-safe），叠加你这里的 fail-closed = 双保险。

### 6.2.2 需要"先审批再执行"？先分清两类审批

很多写操作要"A 申请、B 审批"才能落地（典型：开户、退款、改价）。这里有三种承接方式，选错层会很别扭：

| | 业务接口内部自审批 | 中枢高风险审批 + 业务 webhook | 中枢控制台兜底 |
|---|---|---|
| 谁审批 | 你业务系统里的审核员，在他本来就用的后台审 | 你业务系统/OA/IM 审批流审 | 中枢管理员审 |
| 调用是否先到你的接口 | 会。你的接口收到 AI 请求后自己建业务审批单 | 不会。中枢先冻结工具调用快照，审批通过后才真正调用你的接口 | 不会 |
| 适用 | 业务接口天然就是"申请动作"，例如“提交开户申请” | 高风险真实动作，例如“直接退款/直接改价/删除员工”，希望执行前先让业务审批流确认 | demo、开发联调、无业务审批系统的小团队 |
| 怎么触发 | 你的接口自己返回 `pending_review` | OpenAPI 标 `x-agent-capability.risk.level: high` / `x-agent-capability.approval.required` / `x-agent-capability.approval.when`，路由 `tools.approval.type=business_webhook` | 未配置业务 webhook 时的兜底 |
| 中枢角色 | 传话 + 会话回流 | 冻结 `ApprovalIntent`、投递业务审批、接收 `ApprovalDecision`、批准后重跑并精确放行 | 同左，但由控制台人工裁决 |

#### A. 业务接口内部自审批：接口本身就是“申请”

如果你的业务动作本来就应该先创建申请单，例如“提交开户申请”，让 Agent 调的接口就设计成“提交申请”而不是“直接开户”。这时不要标 `high`，而是让你的接口自己判断并返回待审批：

**第 1 步——你的接口自己把关，返回"待审批"（而不是真执行）：**
```jsonc
// authorize 已通过（A 有"发起开户"的权限）后，你的业务逻辑判定此动作需审批：
// 建一条你自己的审批单（记下 params + 发起人A + 来源会话坐标），然后返回——
// 注意：必须以"受理成功"语义返回（HTTP 200 + ok:true），不要用错误码，否则大脑会当成失败告诉用户。
{ "ok": true, "status": "pending_review", "message": "已提交开户申请，正在等待审核，通过后会第一时间通知你。" }
```
大脑会把 `message` 如实转告用户（"已提交，待审批"），本轮对话到此结束。

**第 2 步——你自己跑审批流**：B 在你后台看到单子、批准。"A 发起、B 审批"全是你的内部逻辑，中枢完全不参与。**发起人 A 是谁**，就来自 §6.2 / §6.2.1 那个已验签的操作主体（你建单时把它记成"发起人"）。

**第 3 步——批准后你自己执行，再把结果回流给用户**：你已经有全部参数，直接执行真实业务逻辑；成功后用 **`POST /send`（形态 D，见 §7）** 把结果推回**原会话**，用户在企微/网页聊天里就收到了，且记进会话历史（大脑后续读得到）。

> **回流到"原会话"需要会话坐标**：中枢调你接口时多带一个头 **`X-Bailing-Conversation: <渠道名>:<收件人>`**（如 `team-im:user_001`）。你把它连同审批单一起存，批准后回填给 `/send` 的 `channel` + `to` 即可。
> - 这个头是**路由提示、不在签名材料里**；**收件人的权威以已验签的 `X-Bailing-On-Behalf-Of` 为准**——建议校验本头里的收件人与主体里的 userid 一致，不一致就用主体的，别用本头的。
> - 网页/H5 入口通常不需要它（用户就在聊天框前等，结果走聊天框回；`/send` 也能推）。它主要服务企微这类"人走了、稍后才批完"的异步场景。

**第 4 步——幂等**：B 可能误点两次、用户可能重复申请。审批单要按你自己的业务键去重，执行一次。

这样审批 100% 在你这边，中枢只负责把"已提交"和"最终结果"两条消息搬进会话。**全程用到的中枢能力都是现成的**（主体透传 §6.2.1 + `/send` 回流 §7），无需中枢为你的审批开任何新功能。

#### B. 中枢高风险审批 + 业务 webhook：执行前冻结真实工具调用

如果 Agent 调的是“真实执行接口”，但某些动作必须先审批，例如退款金额超过 1000、删除员工、批量改价，就在 OpenAPI 里声明风险或参数级确认，并在路由工具配置里选择业务侧审批 webhook：

```jsonc
{
  "sources": [{ "provider": "你的工具源名", "allow": ["tenant.staff.*"] }],
  "approval": {
    "type": "business_webhook",
    "url": "https://你的业务系统/ai/approvals"
  }
}
```

中枢命中高风险调用后会给你 POST `tool_approval_request`，里面有完整 `intent` 和 `decision_contract`。你在自己的审批页/OA/IM 完成审批后，回调：

```http
POST <中枢地址>/approvals/123/decision
Authorization: Bearer <触发方 client token>
Content-Type: application/json

{
  "kind": "tool_approval_decision",
  "schema_version": "bailing.approval-decision.v1",
  "approval_id": 123,
  "job_id": "...",
  "request_id": "...",
  "args_hash": "...",
  "decision_id": "oa:approval:9001",
  "decision": "approved",
  "approver": "user_2002",
  "comment": "确认处理"
}
```

中枢会复核 `approval_id/job_id/request_id/args_hash` 与冻结快照一致；批准后自动重跑原任务，并且只放行 `tool + args_hash` 完全一致的那次调用。完整字段说明见 `CONTRACT.md §2.4`。

### 6.3 发布接口清单 + 后台注册

1. 把你标好注解的清单发布到固定 URL（推荐）：
   ```
   https://<你的域名>/.well-known/bailing/tools.json
   ```
   用 SDK 的 `SpecServer` 一行就能挂出来；或手写一份只含 AI 接口的 spec。
2. 「**工具源**」→ 注册：填 **base_url**（你接口的源站直连地址）+ **签名密钥**（§6.2 那个 secret）+ **spec**（粘贴或填上面的 URL 并开自动刷新）。
3. 「工具清单」预览能看到哪些接口进来了、哪些被跳过及原因。页面右上「**开发文档**」会跳转到官网对应文档。

### 6.4 路由挂工具 + 风险操作走审批

1. 回「触发路由」，给路由加 `tools`：
   ```jsonc
   { "sources": [{ "provider": "你的工具源名", "allow": ["tenant.staff.*"], "subject_field": "operator_uid" }], "max_calls": 5 }
   ```
   - `allow` 是 **scope 白名单**（双闸：你标了 `x-agent-capability.enabled` ∩ 路由 `allow` 允许，才暴露给 Agent）。
   - `subject_field` 指明 metadata 里哪个字段是操作主体（业务后端触发时写入，全程不经 LLM）。
2. **高风险审批承接**：接口标 `high` / `confirm-required` / `confirm-when` 的调用会先冻结成 `ApprovalIntent`。生产推荐路由配置 `tools.approval.type=business_webhook` 投给你的业务审批系统；未配置时才回落到中枢后台「审批意图」页兜底。批准后任务自动重跑，且只放行被批准的那一次调用快照。

### 6.5 几个最容易踩的坑（接工具源前必看）

| 症状 | 原因 | 解法 |
|---|---|---|
| 所有调用都 401 | 服务器**时钟偏移**（最高频）/ 验签实现没对齐 `sha256=` 构造 | 先 `ntp/chrony` 对时；用 `docs/examples` 参考跑冻结向量自检；看 `Verify::failureReason()` 区分 |
| 401 且签名串对不上 | base_url **带了路径前缀**，或端点走了 `?i=&c=&a=` 这类带路由 query 的入口 | base_url 用**纯源站**；端点挂**干净固定路径**；或用 `Verify::currentRequest($secret, '/spec里的path')` |
| `.well-known` 返回 404 或被公开直出（宝塔/BT 面板） | 宝塔默认 vhost 的正则 location 抢了 `.well-known` | 加 `location ^~ /.well-known/bailing/ {...}`（ThinkPHP 要带 `?s=$1`）；**或干脆把 spec_url 换成非点开头路径**如 `/bailing/tools.json` |
| 验签时好时坏 | 从 `$_GET` decode 后重组 URI 验签（改了编码/顺序） | 必须用**原始 `REQUEST_URI`** 验签，或用 SDK 传 spec path 的形式 |

## 7. 形态 D：业务主动经渠道把消息推给用户（进会话历史）

**场景举例**：工单处理完了 → 你的后端要主动经企业微信告诉用户"您的工单已处理完成"。而且——这条推送之后，用户很可能直接在企业微信里追问"好的，什么时候能取？"。

这正是 `/send` 和"自己拿企微 API 发一条"的本质区别：**`/send` 会把这条推送作为「回复方」消息记进"用户在该渠道的那段会话"**。所以用户追问那一轮，渠道背后的大脑读得到上文，不会一脸懵。自己裸发则是一段会话外的失忆。

### 7.1 后台配置（点几下）

1. 先有一个**渠道**（「渠道」页，如企业微信），它绑定了一条路由（大脑）——用户从这个渠道发消息进来本来就走这条路由回复。出站推送会落到同一段会话。
   - ⚠️ 渠道的 `corpid` **必须填**：被动回复能从来访消息现推，但主动推没有来访消息可推导，缺了会报"缺出站凭证"。
2. 「**接入方**」→ 编辑你的 token → **「可推渠道白名单」勾上该渠道**（留空 = 禁止主动推，默认就是禁止）。

### 7.2 业务后端调用：`POST /send`

```bash
curl -X POST "<中枢地址>/send" \
  -H "Authorization: Bearer <你的接入方token>" \
  -H "Content-Type: application/json" \
  -d '{
    "request_id": "myshop-notify-10086",   // 幂等键！同 request_id 不重发（失败要重试请换新 id）
    "channel": "<渠道名>",                   // 后台「渠道」里那个名字
    "to": "<收件人在该渠道的原生 id>",        // 企微即 userid
    "text": "您的工单 #10086 已处理完成"       // 正文原样投递（不经 LLM 改写），≤ 2000 字
  }'
```

**带图片 / 附件（URL 制，可选）**：再加 `images` / `files`，中枢拉取这些 URL → 上传到该渠道 → 投递（企微每个图片/文件各为一条消息，文字在前）。`text` 可省（纯发图也行）。

```bash
curl -X POST "<中枢地址>/send" \
  -H "Authorization: Bearer <你的接入方token>" -H "Content-Type: application/json" \
  -d '{
    "request_id": "myshop-notify-10087",
    "channel": "<渠道名>", "to": "<收件人id>",
    "text": "您的发货单与发票如下",
    "images": ["https://你的域名/发货单.jpg"],
    "files":  [{"url": "https://你的域名/发票.pdf", "name": "发票.pdf"}]
  }'
```

- URL 必须是**中枢能访问到的 http(s)**；图片 ≤10MB、文件 ≤20MB，`images`/`files` 各最多 9 个。
- 附件也会记进会话历史（图片记为 markdown 图、文件记为链接）。
- 任一附件拉取/上传失败 → 整条 `502`、一条都不发（不会半截投递）。

**同步返回**（已投递完才回，不是排队）：

```jsonc
// 成功
{ "ok": true, "job_id": "uuid", "request_id": "myshop-notify-10086", "thread_id": 123 }
// 渠道侧投递失败（凭证缺失 / 收件人非法 / 企微报错）
{ "ok": false, "job_id": "uuid", "request_id": "...", "error": "企微 errcode=..." }
```

关键纪律：
- **正文原样送达**，中枢不改写——单号/金额/时间这类精确内容不会被 AI 动。要 AI 生成措辞是另一种形态（暂未开放），别指望 `/send` 帮你润色。
- **先投递、成功才进历史**：投递失败不会在会话里留下"用户没收到的消息"。
- 记成「回复方」轮次——大脑下一轮看到的是"系统先对用户说了 X"，逻辑正确。
- **没授权该渠道 → 403**；**超限速 → 429**（同 `request_id` 重试不重发）；**渠道侧投递失败 → 502**，按需重试（换新 `request_id`）。
- 在「任务」里能看到每一条推送（含成功/失败、推给谁）。

**企微卡片（textcard，可选）**：加 `card` 对象即推卡片（标题+描述+「查看详情」按钮+免登深链），仅对企微类渠道生效。`text` 仍要给——它作为**不支持卡片渠道的降级文案**，也是**入会话历史**的内容（卡片本身不入历史正文）。

```jsonc
{
  "request_id": "...", "channel": "<渠道名>", "to": "<UserID>",
  "text": "工单 GD2026xxx 已转派给你",              // 降级文案 + 入历史
  "card": {
    "type": "textcard",
    "title": "[待处理] 工单 GD2026xxx 已转派给你",
    "description": "<div class=\"highlight\">客户：xxx</div><div class=\"normal\">…</div>",  // 支持企微限定 HTML
    "url": "https://open.weixin.qq.com/connect/oauth2/authorize?...",                      // 免登深链
    "btntxt": "查看详情"                              // 按钮文案 ≤4 字，缺省「详情」
  }
}
```

**一次发多人（可选）**：`to` 传**数组**或 `"A|B|C"`，中枢用渠道原生能力**合并成一次调用**（企微一次最多 1000 人）——批量通知（任务池 / SLA）首选，省 N 次往返、且**只占一次限速**。每个收件人各自的会话历史都会记上这条（谁回复大脑都接得上）。

```bash
-d '{"request_id":"...","channel":"<渠道名>","to":["UserA","UserB","UserC"],"text":"任务池有新工单待认领"}'
```

**限速**：按你这个接入方的「限速/分」（控制台「接入方」可改，`0` = 不限），且 `/send` 与 `/run`、`/kb/search` **共用同一个按接入方的滑动窗口桶**。批量通知用上面的**多收件人**即可——一批人一次调用、只占一格，基本不会撞 429。若仍要循环单发且量大，把该接入方限速调高或设 `0` 即可（无需单独的桶）。

默认 `llm` 已经能干"理解 + 调工具"。只有当你想让**外部执行器或本地智能体**（自研 agent / 任意脚本 / 其他模型运行时）来干重活时，才需要：

1. 「调度目标」注册一个 `executor` 类目标。
2. 优先在路由的「调用代码 → 执行器接入」复制短引导交给 Agent。完整、版本化的机器说明位于 `<中枢地址>/connect/skills/connect-bailinghub-executor/SKILL.md`，支持原生 Skill，也可作为普通 Markdown 操作手册读取。
3. 手工接入时，在你的机器上跑通用执行器（出站长轮询认领，内网机器即可）：
   ```bash
   curl -fsSL <中枢地址>/connect/executor.mjs -o bailing-executor.mjs
   read -rsp 'BailingHub executor token: ' BAILING_EXECUTOR_TOKEN && printf '\n'
   export BAILING_EXECUTOR_TOKEN
   node bailing-executor.mjs --hub <中枢地址> --targets <目标名> --cmd '<你的命令>'
   ```
   约定只有三条：**stdin 进任务、stdout 出结果、退出码非 0 = 失败**。`--cmd './my-agent.sh'` 可以换成任何能读 stdin、写 stdout 的命令。

   OpenClaw 已配置好模型 Provider 时，可直接使用官方 CLI 适配器：

   ```bash
   curl -fsSL <中枢地址>/connect/openclaw-stdio.mjs -o bailing-openclaw.mjs
   node bailing-executor.mjs --hub <中枢地址> --targets <目标名> \
     --runtime openclaw --cmd 'node bailing-openclaw.mjs --agent bailinghub-executor'
   ```

   适配器会把中枢会话映射为 OpenClaw 会话，并只把最终文本写入 stdout。默认不会把 `BAILING_TOOL_TOKEN` 等业务工具凭据转交给 OpenClaw；先验证任务领取与回传，再单独配置受治理的工具调用。

## 9. 多轮会话怎么办

- 一问一答：路由 `session_policy=new`，不用管上下文。
- 要按业务键续聊（同一工单多轮追问）：路由 `session_policy=per_key`，触发时 `metadata` 带业务键即可——**中枢自动从对话总账装配上下文，你不用自己存历史**。
- 网页聊天组件：同访客自动续；要平行多个会话用 `thread_id`（见 `CONTRACT.md §1.1`）。

## 10. 联调自测清单

接完后按这个顺序验，每步通过再下一步：

- [ ] **登录后台**，「模型凭证」有可用凭证。
- [ ] **形态 A**：`curl` 调 `/run` 返回 `202 + job_id`；几秒后 `GET /jobs/<id>` 看到 `status=done` 和 `report`。
- [ ] 故意用**同一个 `request_id`** 再调一次 → 返回**同一个 job_id**（幂等生效）。
- [ ] 填了 `callback_url` 的话：你的回调端点收到 POST，且**验签通过**（故意改一字节应 401）。
- [ ] **形态 B**：网页贴 `<script>` 后聊天框出现、能问能答；从**别的 Origin** 打开应被白名单拦。
- [ ] **形态 C**：「工具清单」能看到你的接口；让 Agent 问一句触发它；故意传**非法主体**应被你的 authorize **403**（验证授权闸真的生效，不是 `return true`）。
- [ ] **形态 D**：`curl` 调 `/send` 收到 `ok:true`，目标用户在渠道里收到消息；让该用户回一句，确认大脑接得上刚推的内容（历史连续）；没授权的渠道应 **403**。
- [ ] **降级演练**：把中枢地址改错/超时，确认你的业务**照常走、不报 500**。

## 11. 必须遵守的降级约定

这是"消费关系"的本质——**单向、松、可断**：

- 中枢无响应 / 超时 / 503：业务**不阻塞、不报 500**，原有流程照常走，事后可重试**同 `request_id`**。
- 报告解析失败：`status=done` 但 `report.needs_human=true`，交人工。
- 永远不要让"中枢挂了"变成"我的业务挂了"。
