# 社区衍生与生态合作

百灵中枢欢迎开发者基于开源核心创建独立发行版、行业适配、执行器、渠道适配器和其他集成，也欢迎不使用百灵中枢代码、但遵循 ACC 的独立实现。

社区项目不需要成为百灵中枢的一部分，也不需要把所有修改贡献回上游。我们尊重独立名称、产品方向、维护团队和治理方式。对具备通用价值的改进，我们欢迎通过 Issue、Discussion 或 Pull Request 一起完善上游。

## 可以如何参与

| 方式 | 适合的项目 |
|---|---|
| 上游贡献 | 对所有用户都有价值的缺陷修复、契约完善、文档、测试和通用能力。 |
| 社区发行版 | 面向特定行业、部署环境或用户体验的独立 BailingHub 衍生版本。 |
| 生态组件 | 模型、执行器、渠道、存储、业务系统连接器、SDK 和部署面板适配。 |
| ACC 独立实现 | 不依赖 BailingHub 代码、独立实现 ACC 契约的运行时或工具。 |

## 官方展示原则

当出现符合条件的社区项目后，百灵中枢会建立公开生态页面，为优秀的独立项目提供可发现入口。申请展示的项目应当：

- 提供公开可访问的项目、文档或产品页面；
- 清楚说明维护者、独立身份以及与 BailingHub 或 ACC 的关系；
- 遵守所使用代码和依赖的许可证、NOTICE 与归属要求；
- 说明兼容的 BailingHub 或 ACC 版本，以及主要差异和适用场景；
- 具备真实维护内容，不是只替换名称、Logo 或截图的重复发布；
- 不暗示百灵中枢官方提供安全认证、商业担保或维护责任。

官方展示意味着“这是值得社区发现的独立项目”，不等于官方认证、质量担保、商业背书或安全审计。项目维护者始终对自己的发行版、服务和用户负责。

## 名称与来源

社区发行版应使用自己的名称和视觉标识。可以如实说明“基于 BailingHub”“兼容 BailingHub”或“实现 ACC”，但不得让用户误以为它是百灵中枢官方发行版，或已经获得未经明确授予的官方认证和背书。

具体代码使用和再分发条件以 [LICENSE](../LICENSE) 与 [NOTICE](../NOTICE) 为准。名称和标识不因 Apache License 2.0 自动获得授权。

## 我们希望形成的关系

我们不把社区衍生视为需要阻止的竞争。优秀的独立项目可以探索不同的行业、交付方式和产品方向；通用能力可以回到上游，独立能力可以继续由社区维护。百灵中枢愿意公开优秀实现、保持兼容对话，并在双方认可时开展文档、适配和发布协作。

当前可通过 [Feature request](https://github.com/bailinghub/bailinghub/issues/new?template=feature_request.yml) 介绍你的项目或合作设想。首个符合条件的社区项目出现后，我们会建立正式的生态展示与提交入口。
