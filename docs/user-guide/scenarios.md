# 场景化配置指南

这页按业务目标来写。你可以先找到最接近的场景，再按步骤配置。

每个场景都分成四部分：

- 目标：你想实现什么。
- 后台怎么配：使用者在中枢后台做什么。
- 配完拿到什么：可以交付给开发者或业务侧什么。
- 开发者看哪里：后续技术对接文档。

## 场景一：先跑一个只读业务查询助手

### 目标

让 Agent 能根据用户问题查询业务数据，比如：

- 查订单。
- 查会员。
- 查工单。
- 查库存。
- 查设备状态。

这是最适合作为第一阶段试点的场景，因为它只读、风险低、业务价值清晰。

### 后台怎么配

1. 在「模型凭证」里配置对话模型，并点“验证”。
2. 在「工具源」里注册业务系统提供的只读接口 spec。
3. 在「触发路由」里新建一条路由，例如“订单查询助手”。
4. 路由目标选择 `llm`。
5. 路由 `tools.sources[]` 添加刚才的工具源；需要跨系统查询时可继续添加其他工具源。
6. 每个来源的 `allow` 只放行查询相关 scope，例如 `order.read` 或 `order.*.read`。
7. 如果查询必须基于登录用户权限，设置 `subject_field`，例如 `operator_uid`。
8. 在「接入方」里创建业务系统接入方，允许调用这条路由。

### 配完拿到什么

- 路由 key。
- 接入方 token。
- `/run` 调用代码。
- 工具源签名密钥。
- 查询接口需要的 `subject_field`。

### 给开发者怎么说

你可以这样交付：

```text
我们要先接订单查询助手。
中枢路由：order-query
接入方：mall-server
业务侧触发时调用 /run，metadata 里带 operator_uid。
工具源会调用你们的订单查询接口，请按工具源密钥实现 sha256 验签。
订单接口仍然用你们自己的权限表判断 operator_uid 能看哪些订单。
```

## 场景二：做一个网页售后助手

### 目标

在业务网页或管理后台里嵌入一个聊天组件，让用户可以问售后问题，并让 Agent 能查订单、创建工单。

### 后台怎么配

1. 「模型凭证」配置对话模型。
2. 「知识库」上传售后政策、退款规则、常见问题。
3. 「工具源」注册业务系统接口：
   - 查询订单：只读。
   - 创建工单：中风险或低风险，取决于业务。
   - 申请退款：通常中风险或高风险。
4. 「触发路由」新建“售后助手”。
5. 路由绑定知识库。
6. 路由绑定工具源，只放行售后相关 scope。
7. 路由设置会话策略：
   - 如果按访客延续，用聊天入口默认 visitor。
   - 如果按登录用户延续，使用票据 uid。
8. 「聊天入口」新建入口，绑定售后助手路由。
9. 配置外观、开场白、Origin 白名单。
10. 如需登录身份，配置票据签发方。
11. 复制嵌入代码给前端开发者。

### 配完拿到什么

- 聊天入口 `entry_key`。
- `<script>` 嵌入代码。
- 路由 key。
- 票据签发说明。
- 工具源签名密钥。

### 给开发者怎么说

```text
请把这段聊天组件 script 嵌入售后页面。
如果用户已登录，请后端按中枢票据规则签发 ticket，前端放到 data-ticket。
业务工具接口要实现中枢工具调用验签。
Agent 创建工单时，接口仍按 ticket 里的 uid 做业务权限判断。
```

## 场景三：让 Agent 发起退款，但不能直接退款

### 目标

Agent 可以理解用户诉求、查询订单、发起退款申请，但不能绕过业务审批直接退款。

### 后台怎么配

1. 工具源里声明三个工具：
   - 查询订单：只读。
   - 退款试算：只读或低风险。
   - 发起退款申请：中风险。
2. 如果是“立即退款”接口，通常声明为高风险或 `confirm-required`。
3. 触发路由里只放行退款场景需要的 scope。
4. 路由里配置审批承接方式：
   - 如果业务系统已有审批流，推荐让工具接口只创建退款申请，审批回业务系统处理。
   - 如果确实需要中枢先拦截高风险调用，启用审批意图。
5. 任务页观察审批意图和工具调用记录。

### 配完拿到什么

- 退款相关工具 scope。
- 高风险工具声明。
- 审批意图记录。
- 业务侧审批 webhook 或申请接口说明。

### 给开发者怎么说

```text
Agent 不直接打真实退款接口。
请提供“创建退款申请”接口，Agent 调用后返回 pending_approval。
如果未来要接立即退款接口，必须声明 high 或 confirm-required。
业务侧收到中枢调用后，仍按 operator_uid 判断此人是否有发起退款申请权限。
```

### 开发者看哪里

- [TOOLS_DESIGN.md §3.4 字段心智](../TOOLS_DESIGN.md#34-字段心智声明-ai-触达边界不声明业务审批规则)
- [AI友好工具设计指南.md](../AI友好工具设计指南.md)

## 场景四：做一个内部知识库问答助手

### 目标

让员工问制度、流程、产品资料、售后政策等文档内容。

### 后台怎么配

1. 「模型凭证」配置对话模型。
2. 「模型凭证」配置向量模型。
3. 「知识库」新建知识库。
4. 上传文档或配置数据源。
5. 做命中测试。
6. 「触发路由」新建“员工知识助手”。
7. 路由绑定知识库。
8. 如果只允许内部系统触发，在「接入方」限制路由白名单。
9. 如果要嵌网页，在「聊天入口」绑定这条路由。

### 配完拿到什么

- 知识库 ID。
- 路由 key。
- 聊天入口代码或 `/run` 调用代码。

### 给开发者怎么说

如果只是后台问答，开发者只需要嵌聊天组件或调用 `/run`。如果文档来自业务系统，则需要开发者提供同步文档的数据源。

### 开发者看哪里

- [CONTRACT.md §2.1 知识检索](../CONTRACT.md#21-知识检索post-kbsearch)
- [QUICKSTART.md](../QUICKSTART.md)

## 场景五：业务后端主动触发一个 Agent 任务

### 目标

业务系统在某个事件发生时主动让中枢处理，比如：

- 工单创建后自动分析优先级。
- 客户提交投诉后自动总结。
- 订单异常后自动生成处理建议。
- 晚上定时生成经营分析。

### 后台怎么配

1. 新建触发路由。
2. 配好模型、知识库、工具源和送达方式。
3. 新建接入方，允许调用这条路由。
4. 在路由行点击“调用代码”。
5. 把调用代码交给业务开发者。

### 配完拿到什么

- route key。
- 接入方 token。
- `/run` 调用代码。
- callback 或轮询说明。

### 给开发者怎么说

```text
在业务事件发生后调用中枢 /run。
request_id 请使用你们自己的业务唯一键，保证幂等。
input 放要给 Agent 看的业务事件正文。
metadata 放业务对象 ID、操作主体、会话键。
如果需要结果回业务系统，用 callback_url 或轮询 /jobs/{job_id}。
```

- [CONTRACT.md §1 POST /run](../CONTRACT.md#1-触发post-run)

## 场景六：接入本地模型或 Ollama

### 目标

不使用公有云模型，而是让中枢调用本地或内网部署的模型。

### 后台怎么配

1. 确认本地模型服务提供 OpenAI-compatible API。
2. 在「模型凭证」中新建凭证。
3. base URL 填本地或内网地址，例如 `http://host.docker.internal:11434/v1` 或你的内网模型网关。
4. API key 按模型服务要求填写，没有 key 可填占位值，具体看服务配置。
5. 默认模型填实际模型名。
6. 点“验证”。
7. 在触发路由的 `llm` 目标配置里选择该凭证和模型。

### 配完拿到什么

- 模型凭证名称。
- 可达性验证结果。
- 使用该模型的路由。

### 给开发者怎么说

一般不需要业务开发者感知模型来源。业务仍然调用同一个 route。后续从本地模型换到云模型，只改中枢配置，不改业务调用代码。

### 开发者看哪里

- [QUICKSTART.md](../QUICKSTART.md)
- [CONTRACT.md](../CONTRACT.md)

## 场景七：接入自己的脚本、OpenClaw、Codex 或其他 agent

### 目标

你已经有一个本地脚本、自研 agent、RPA、OpenClaw、Codex、Claude Code、国内智能体工具或第三方运行时，希望它来处理任务，而不是让 `llm` 直接处理。

### 后台怎么配

1. 「调度目标」新建 executor 类目标，例如 `local-agent`。
2. 「执行器」签发执行器令牌，允许认领 `local-agent`。
3. 「触发路由」选择 target=`local-agent`。
4. 开发者在自己的机器上运行执行器命令。

### 配完拿到什么

- target 名称。
- 执行器令牌。
- 中枢地址。
- 执行器启动命令。

### 给开发者怎么说

```text
请在你的机器上启动中枢执行器。
执行器使用出站长轮询认领任务，不要求中枢访问你的内网机器。
你的命令从 stdin 读取任务，向 stdout 输出结果。实际处理器可以是本地脚本，也可以是 OpenClaw、Codex、Claude Code、国内智能体工具或自研 agent 的封装命令。
```

## 场景八：支持图片、语音和文件输入

### 目标

用户除了输入文字，还可能发送：

- 图片：商品图、截图、凭证、故障照片。
- 语音：口述问题。
- 文件：PDF、Word、Excel、CSV。

### 后台怎么配

1. 「媒体存储」确认默认本地存储可用，生产可配置对象存储。
2. 「模型凭证」按需配置视觉、语音、文件处理相关模型。
3. 「触发路由」里配置输入策略：
   - 图片：识别、直送多模态模型、忽略。
   - 语音：转写、直送支持语音的模型、忽略。
   - 文件：抽取文本、摘要、直送执行器、忽略。
4. 「聊天入口」开启附件能力。
5. 用真实图片、语音、文件跑任务页检查 trace。

### 配完拿到什么

- 媒体存储桶名称。
- 路由输入策略。
- 附件 URL。
- 任务 trace 里的识别/转写/抽取记录。

### 给开发者怎么说

如果业务侧只使用中枢聊天组件，开发者通常不用处理上传。若业务系统自己调用 `/run` 并携带附件，需要按 CONTRACT 里的附件输入结构传递。

## 场景九：让 Agent 在某个业务页面里更懂当前页面

### 目标

用户在某个具体页面里打开聊天组件，例如订单详情页、商品详情页、会员详情页，希望 Agent 知道他正在看哪里。

### 后台怎么配

1. 「聊天入口」绑定路由。
2. 在聊天入口的页面登记里配置 URL 模式、页面名、页面说明。
3. 如果页面有稳定业务标识，可让前端传 `page_key`。
4. 路由按需启用页面感知检索。
5. 任务页查看 `page_context` 是否命中。

### 配完拿到什么

- 页面登记规则。
- 页面上下文注入结果。
- 页面相关知识检索偏置。

### 给开发者怎么说

```text
聊天组件会自动上报当前 URL 和 title。
如果你们有更稳定的页面标识，请给组件传 page_key。
page_context 只用于理解和检索，不作为权限依据。
```

### 开发者看哪里

- [CONTRACT.md §1.2 网页聊天增强](../CONTRACT.md#12-网页聊天的两个运行时增强)

## 场景十：上线前做一次检查

### 目标

确认配置能跑通，不要让开发者或业务用户第一次使用就踩基础错误。

### 后台怎么配

1. 进入「系统体检」。
2. 查看配置错误和提醒。
3. 修复错误项。
4. 运行 Smoke。
5. 进入「任务」检查 Smoke 生成的任务。
6. 如有工具源，执行授权探针。
7. 如有模型凭证，执行验证。

### 配完拿到什么

- Smoke 结果。
- 配置体检结果。
- 任务 trace。
- 可交给开发者的错误对象和问题描述。

### 给开发者怎么说

不要只说“Agent 不行”。应该给出：

```text
路由：xxx
任务：job_id / request_id
错误：系统体检里的原始错误
trace：工具调用返回 403 / 模型凭证验证失败 / target 不存在
需要你处理：业务接口验签、权限、模型 key、callback、还是执行器在线
```
