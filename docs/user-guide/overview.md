# 百灵中枢解决什么问题

## 先从一个真实场景说起

假设你有一套业务系统，比如门店管理、订单售后、ERP、CRM、工单系统或内部运营后台。现在你希望 Agent 能帮你做这些事：

- 用户问“我这个订单怎么还没发货”，Agent 能查订单并回答。
- 客服说“帮这个客户建一个售后工单”，Agent 能按规则创建工单。
- 运营问“今天有哪些异常退款”，Agent 能查数据、总结并给出列表。
- 管理员说“把这份制度文档接进去，让员工能问”，Agent 能基于知识库回答。
- 某个高风险动作，比如退款、删员工、改权限，Agent 不能直接乱做，必须有审批和审计。

如果没有百灵中枢，业务系统通常要自己做一整套事情：

1. 接模型：选模型、管 key、写调用代码。
2. 接入口：业务后台、网页聊天、企业微信/钉钉/飞书等渠道消息、第三方系统入口都要自己接。
3. 管上下文：谁在问、在哪个页面问、前面聊过什么、要注入哪些知识。
4. 管工具调用：Agent 要调哪个接口、参数怎么限制、失败怎么记录。
5. 管权限：这个人能不能查、能不能改、能不能删。
6. 管审批：高风险动作谁审核，审核通过后怎么继续执行。
7. 管审计：Agent 做过什么，调用了哪个接口，为什么失败。
8. 管送达：结果怎么回业务系统，怎么推给用户或管理员。

这些事情不是某一个功能点，而是一条完整运行链路。百灵中枢的价值就是把这条链路从业务系统里抽出来，做成一个独立的 Agent 控制面。

## 一句话理解

百灵中枢不是一个单纯聊天机器人，也不是一个固定工作流工具。

它是放在业务系统和 Agent 大脑之间的一层中枢：业务系统把任务交给中枢，中枢根据后台配置决定用哪个大脑、带哪些知识、允许调用哪些业务接口、是否需要审批、结果怎么回去，并把整个过程记录下来。

业务系统不用把 Agent 能力硬编码进自己的业务代码里。它只需要按中枢的契约触发任务、暴露必要工具、接收结果。

## 它适合解决哪些问题

### 1. 老系统想快速拥有 Agent 操作能力

很多传统系统已经有成熟的后台、权限、接口和业务流程，但没有 Agent。百灵中枢不要求你重写系统，只需要把已有接口中适合 Agent 使用的部分声明成工具。

例如：

- 原来人工在后台查订单，现在 Agent 可以调用查订单接口。
- 原来客服点击按钮创建工单，现在 Agent 可以调用创建工单接口。
- 原来管理员看报表后总结，现在 Agent 可以调用只读接口并生成摘要。

Agent 做的是“替人发起已有业务动作”，不是绕过业务系统新开一套权限。

### 2. 多个 Agent 场景需要统一治理

一个业务系统里可能会有多个 Agent 场景：

- 售后助手
- 经营分析助手
- 员工知识库问答
- 运营配置助手
- 网页客服组件
- 企业微信/钉钉/飞书等渠道助手

如果每个场景都单独接模型、单独写权限、单独做审计，后期会很难维护。百灵中枢用“触发路由”把这些场景统一起来：每个路由代表一个业务场景，路由里配置目标大脑、工具白名单、知识库、记忆、送达和预算。

### 3. Agent 可以调用业务接口，但不能失控

Agent 一旦能调业务接口，就必须回答几个问题：

- 哪些接口允许 Agent 看见？
- 哪些接口只能查询，哪些会改变业务状态？
- 哪些接口需要审批？
- Agent 调用接口时，替哪个业务用户操作？
- 业务侧怎么确认请求确实来自中枢？
- 失败了怎么排查？

百灵中枢把这些问题拆成两层：

- 中枢管“Agent 能够到哪里”：路由白名单、工具白名单、风险等级、限流、审计、审批意图。
- 业务系统管“这个人此刻能不能做”：业务侧验签后，继续用自己的权限表和审批逻辑判断。

这能避免一个常见误区：不是把业务权限搬到中枢里重写一遍，而是让 Agent 调用也走业务系统原来的权限裁决路径。

### 4. 需要能追溯、能解释、能定位问题

Agent 系统上线后，最常见的问题不是“完全不能用”，而是：

- 为什么这次回答不对？
- 它到底看了哪些知识？
- 它有没有调用业务接口？
- 调用参数是什么？
- 业务接口返回了什么？
- 是模型问题、配置问题、工具问题，还是权限问题？

百灵中枢的“任务”和“执行轨迹”就是为这个准备的。每一次触发都会形成一条任务记录，能看到路由、输入、上下文装配、工具调用、审批、送达、最终结果和错误原因。

### 5. 想先小范围试点，再逐步放开

中枢适合从一个小场景开始：

1. 先做只读查询，比如查订单、查工单、查知识库。
2. 再做低风险动作，比如创建申请、生成草稿、打标签。
3. 最后才考虑高风险动作，比如退款、删除、权限变更，并接入审批。

每一步都可以在后台配置路由和工具白名单，不需要一次性把所有能力开放给 Agent。

## 它不解决什么

百灵中枢不会替业务系统做这些决定：

- 某个业务用户到底有没有退款权限。
- 某个门店员工能不能查看另一个门店的数据。
- 某个订单是否满足退款条件。
- 某个审批应该由哪个业务主管处理。

这些仍然属于业务系统自己的领域。中枢会把“替谁操作”的主体、任务号、签名、工具参数和审批意图带过去，但最终业务规则仍由业务系统决定。

## 开源版的部署边界

开源版按单组织部署设计。你可以在一套中枢里配置多个业务系统、多个接入方、多个路由、多个模型和多个工具源，但它们共享同一个后台、同一套管理员和同一套审计空间。

典型使用方式：

- 一家公司部署一套百灵中枢，给自己的多个业务系统接 Agent。
- 一个团队部署一套百灵中枢，先服务几个内部场景。
- 一个项目部署一套百灵中枢，作为该项目的 Agent 控制面。

如果你要让多个互不信任的组织共用同一套中枢，不建议用开源版直接混在一起；更简单、更安全的方式是每个组织独立部署一套。

## 接入前先问这五个问题

在配置后台之前，先把业务需求拆清楚：

1. 用户从哪里发起：业务后台、网页聊天、企业微信/钉钉/飞书等渠道消息，还是后端事件？
2. Agent 要做什么：回答问题、查资料、查业务数据、创建业务对象，还是执行高风险动作？
3. 它需要哪些业务接口：只读查询、创建申请、真实执行、批量操作？
4. 它替谁操作：匿名访客、登录用户、员工、管理员，还是系统任务？
5. 结果要去哪里：用户等回答、业务系统接 callback、后台查看任务，还是推送到渠道？

这五个问题对应中枢里的五类配置：入口、路由、工具源、身份主体、送达方式。

## 和开发者怎么分工

使用者负责定义业务心智：

- 哪些场景先做。
- 哪些接口可以开放给 Agent。
- 哪些动作属于高风险。
- 哪些知识库需要接入。
- 哪些页面要嵌聊天组件。
- 结果应该如何回到业务流程。

开发者负责落地技术接入：

- 按中枢的调用代码触发 `/run`。
- 暴露工具源 OpenAPI 或使用 SDK 标注工具。
- 实现业务侧验签和授权。
- 接收 callback 或处理审批回传。
- 在网页里嵌入聊天组件。

百灵中枢的后台配置就是两者之间的沟通界面：使用者在后台定义场景，开发者拿到对应的 token、代码、URL、签名密钥和文档去对接。
