# 核心概念：把业务需求拆成中枢配置

这一页不讲 API 细节，只讲后台里的核心对象之间是什么关系。

## 总体关系

可以把百灵中枢理解成下面这条链路：

```text
入口
  └─ 把用户问题、业务事件或外部消息送进来
      ↓
触发路由
  └─ 判断这是哪个业务场景，应该交给哪个大脑，带哪些能力
      ↓
调度目标
  └─ 真正处理任务的大脑：llm、本地模型、执行器、第三方 agent
      ↓
上下文
  └─ 历史对话、知识库、页面上下文、图片/语音/文件理解
      ↓
工具源（可多个）
  └─ Agent 可以跨业务系统安全调用的接口清单
      ↓
审计与送达
  └─ 记录发生了什么，并把结果回到业务侧或用户侧
```

后台里的菜单基本都围绕这条链路展开。

## 触发路由：一个业务场景

触发路由是中枢最核心的配置。你可以把一条路由理解成“一个 Agent 场景”。

例如：

- `after-sales-assistant`：售后助手。
- `order-query`：订单查询助手。
- `operation-analyst`：经营分析助手。
- `staff-knowledge`：员工制度问答。
- `refund-review`：退款处理助手。

一条路由通常决定这些事：

- 交给哪个调度目标处理。
- 使用哪个模型凭证或目标配置。
- 是否按某个业务字段延续会话。
- 是否注入知识库。
- 是否启用对话记忆。
- 是否允许调用某个工具源里的部分工具。
- 结果是否需要 callback 或渠道送达。
- 高风险动作是否需要审批。
- 是否有预算、限流或重试策略。

路由不是接口，也不是模型。路由是把一个业务场景的 Agent 运行规则集中起来。

## 接入方：哪个业务系统能调用哪些路由

接入方是一把给业务系统的钥匙。

一个业务系统要调用中枢，通常先在后台创建一个接入方。接入方会得到：

- `app_id`：接入方标识。
- `token`：业务系统调用中枢时使用的密钥。
- `allowed_routes`：这个业务系统允许调用哪些路由。
- `rate_limit_per_min`：每分钟调用限制。

接入方解决的是“谁可以触发中枢”。

它不等于最终业务用户。最终业务用户通常由业务系统在 `metadata.principal` 或 `metadata` 的主体字段里声明。例如业务系统用自己的登录态确认当前用户是 `u_1001`，再把这个主体传给中枢。

## 调度目标：任务交给谁处理

调度目标是大脑插座。路由最终会指向一个调度目标。

常见目标：

- `llm`：中枢进程内调用云端模型或 OpenAI-compatible 模型服务。
- 自定义 executor 目标：由你的本地机器、内网脚本、自研 agent、OpenClaw、Codex、Claude Code、国内智能体工具或第三方 agent 出站认领任务。
- demo-agent：开源 demo 里的确定性演示目标，用来不依赖真实模型跑通工具治理闭环。

如果你的需求只是“让 Agent 回答、查知识库、调用工具”，通常先用 `llm`。如果你有自己的本地 agent、私有环境脚本、长任务处理器，或者希望把 OpenClaw、Codex、Claude Code、国内智能体工具接到中枢里统一调度，就注册 executor 类目标。

## 模型凭证：中枢怎么调用模型

模型凭证记录模型服务地址、API key、默认模型和用途。

它可以用于：

- 对话模型：回答问题、做工具编排。
- 视觉模型：理解图片。
- 语音模型：把语音转文字，或直送支持语音的模型。
- 向量模型：知识库或工具检索。

模型凭证不是业务场景。业务场景在路由里配置，路由再引用对应模型能力。

## 工具源：Agent 能调用哪些业务接口

工具源是业务系统暴露给 Agent 的接口清单。

业务系统通过 OpenAPI 或 SDK 声明：

- 哪些接口允许 Agent 调用。
- 每个接口叫什么。
- 参数是什么。
- 这个接口是只读、低风险、中风险还是高风险。
- 是否需要主体。
- 是否需要确认或审批。

中枢注册工具源后，不会自动把所有工具都交给 Agent。还需要在路由的 `tools.sources[]` 中选择一个或多个来源，并分别配置 `allow`，只放行当前场景需要的 scope。不同来源的 `operationId` 必须在同一路由内保持唯一，避免调用被路由到错误系统。

工具源解决的是“业务能力如何被 Agent 安全触达”。

## 知识库：Agent 回答时能参考哪些资料

知识库用于把文档、制度、帮助中心、产品说明、FAQ 等内容变成可检索资料。

知识库不会自动影响所有路由。只有路由里配置了知识库，任务执行时才会检索并注入相关片段。

适合放进知识库的内容：

- 使用手册。
- 售后政策。
- 员工制度。
- 产品说明。
- FAQ。
- 内部流程文档。

不适合只靠知识库解决的内容：

- 实时订单状态。
- 当前库存。
- 用户权限。
- 需要写入或变更业务状态的操作。

这些应该走工具源。

## 聊天入口：把路由嵌到网页里

聊天入口是一个公开的网页组件入口。它绑定一条路由，然后给你一段 `<script>` 嵌入代码。

适合：

- 官网客服。
- 业务后台里的 Agent 助手。
- H5 页面里的问答入口。
- 某个功能页的上下文助手。

聊天入口解决的是“用户从网页上怎么和这条路由对话”。

如果需要让登录用户带可信身份，业务后端应签发访客票据，组件带票进入中枢。中枢只验证票据，真正的权限仍由业务系统在工具接口里裁决。

## 入站渠道：让外部平台消息进中枢

入站渠道用于接企业微信、钉钉、飞书等外部平台消息。渠道配置绑定一条路由，外部消息进来后就按这条路由处理。

它和聊天入口的区别：

- 聊天入口面向网页组件。
- 入站渠道面向外部消息平台。

## 媒体存储：图片、语音、文件先放哪里

当用户通过聊天入口或渠道发送图片、语音、文件时，中枢需要先保存这些素材，再根据路由策略决定怎么处理。

默认可以使用服务器本地存储。生产环境如果需要 CDN、多副本共享、生命周期管理或已有对象存储，可以在后台配置媒体存储桶。

媒体存储只负责“素材落地和形成 URL”。具体是识别图片、转写语音、抽取文件文字，还是直送给多模态模型，由路由的输入策略决定。

## 任务：一次 Agent 运行记录

每次业务系统调用 `/run`、用户在聊天入口提问、外部渠道收到消息，都会形成一个任务。

任务里可以看到：

- 输入内容。
- 命中的路由。
- 调度目标。
- 会话信息。
- 工具调用。
- 知识引用。
- 审批意图。
- 最终输出。
- 错误信息。

任务是排查问题的第一入口。

## 审批意图：Agent 想做高风险动作时先冻结

如果工具声明为高风险，或配置了确认条件，中枢不会直接外发调用，而是生成审批意图。

审批意图包含：

- Agent 想调用哪个工具。
- 参数快照是什么。
- 替哪个主体操作。
- 风险和原因。
- 审批通过后只能执行这一份参数快照。

中枢可以记录审批意图，但业务侧通常更适合承接真正的审批流程。比如退款审批应该回到订单系统或财务系统里，由原有主管审批链处理。

## 变更审计和执行轨迹：为什么这样回答

中枢有两类记录：

- 变更审计：谁改了配置、工具源、路由、凭证等。
- 执行轨迹：某次任务运行时发生了什么。

前者解决“后台配置为什么变了”，后者解决“这次 Agent 为什么这样做”。

## 一个需求如何拆成配置

如果你的需求是“让门店后台有一个售后 Agent 助手”，可以拆成：

- 入口：网页聊天入口，嵌入门店后台。
- 路由：售后助手路由。
- 调度目标：`llm`。
- 模型凭证：选择一个对话模型。
- 知识库：售后政策、退款规则、商品说明。
- 工具源：查订单、创建售后工单、申请退款。
- 接入方：门店后台服务端，用来签发访客票据或触发 `/run`。
- 审批：退款申请或高金额退款进入业务审批。
- 任务：上线后用任务页观察回答和工具调用。

这就是中枢的基本使用方法：先把业务需求拆成入口、路由、大脑、知识、工具、身份、送达和审计，再分别配置。
