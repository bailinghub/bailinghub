# 百灵中枢 · 兼容性与升级策略

> 这份策略用于公开发布后的版本演进。目标是让绝大多数升级都能平滑完成：拉新版、执行数据库结构同步、重启 hub；已配置的路由、凭证、工具源、聊天入口不需要重配。破坏性变更必须显式说明，并提供过渡窗口。

---

## 一、升级动了哪些面

部署方拿到的是整套 `bailinghub`（同一 tag）。一次升级可能触及下列面，每个面都有各自的"不入侵"机制（见第三节）：

| 面 | 是什么 | 升级时怎么不被入侵 |
|---|---|---|
| **hub 服务**（`src/`） | 中枢进程本体 | 拉新版重启即可；无状态逻辑 |
| **官方 SDK**（`sdk/php`·`sdk/php7` + `docs/examples/*`） | 业务系统里嵌的验签/建 spec 代码 | 仅当某次动了 **wire 格式**才需同步更新；有第四节硬约束兜底 |
| **DB 状态库**（`sql/*.sql`） | 路由/凭证/工具源/聊天入口/渠道/页面登记表/知识库/限速账本等**全部运行时配置** | 结构变更账本 + 只增不毁（§三）；`db:init` 幂等可反复跑，**绝不重建已存在的表** |
| **brain 文件配置**（`brain/`） | executor 大脑的能力档 / agent 提示词 / runbook | `.local` 叠加层（§三）：定制写 `.local` 文件，升级只动默认文件、永不冲突 |
| **嵌入组件**（`web/widget/widget.js`） | 别人网站里 `<script>` 引的聊天组件 | 受 `WIDGET_API` 约束的稳定嵌入面（§六）；hub 升级=嵌入页自动拿新版，故只增不改 |

> 关键心智：**配置不在代码里**。路由、凭证、工具源、聊天入口全在 DB（`bz_` 表）、由控制台配；`.env`/密钥管理器/`brain/*.local.*` 是部署方本地材料、已 gitignore。`config.json` 只适合本地开发或非敏感结构配置；生产部署设置 `BAILING_ENV=production` 后，敏感项必须由环境变量或密钥管理器注入。

## 二、版本语义（SemVer）：看一眼版本号就知道升级有多重

| 版本位 | 含义 | 部署方要做什么 |
|---|---|---|
| **patch**（x.y.**Z**） | 修 bug、补文档，**零 wire / 零 schema 变更** | 拉新版重启 hub，完事 |
| **minor**（x.**Y**.0） | **只增不改**：新可选头 / 新 `x-agent-capability` 注解 / 新信封字段 / 新能力 / 新数据库表或字段 | 拉新版 → `db:init` → 重启 hub；现有业务代码照跑，新能力想用再用 |
| **major**（**X**.0.0） | 含**破坏性变更**：改签名构造/算法、删字段、改语义、改 schema 列含义、bump `WIDGET_API` | 读 CHANGELOG 的 `BREAKING` 段；按第四节过渡窗分步升级，**不强制 hub 与业务系统/SDK 同时切** |

> DB schema 与嵌入契约都吃这套语义：加表/加列/加可选属性 = minor、drop-in；删列/改列义/改属性名 = major、走过渡窗。

## 三、为什么 minor 升级几乎零入侵：六个内建机制

**wire 面（三个）：**

1. **签名标签 `sha256=` 是扩展点，不是版本号**。要换签名算法/构造，就**另起一个自描述新标签**（如 `sha256-2=`），绝不复用已有标签语义，也不在协议里堆隐式版本。验签侧靠前缀分派，新增标签天然不冲突。
2. **spec 的 `x-agent-capability` 未知字段一律忽略**（CONTRACT §2.4a）。注解注册表加新键时，未升级的 hub 会直接跳过不认识的键——业务按自己节奏采用新注解，不被新版本逼着改。
3. **信封新增字段都可选**。回调/送达/job 对象里加字段，既有部署方读不到也不报错（JSON 多字段无害）。所以"加能力"永远是 minor。

**部署面（三个）：**

4. **数据库结构有账本、只增不毁**。`db:init` 维护 `bz_schema_migrations` 账本：已应用的 `sql` 文件直接跳过、永不重放；新版本只应用新增的编号文件。`CREATE TABLE IF NOT EXISTS` + 只 `ADD COLUMN ... DEFAULT` 保证**绝不重建/清空已配置的表**。详见 `sql/README.md`。
5. **brain 文件走 `.local` 叠加层**。`brain/profiles.json`·`agents/*.md`·`runbooks/*.md` 是仓库默认；部署方定制写进同名 `.local` 文件（已 gitignore）。升级 `git pull` 只动默认文件、永不碰 `.local`——**默认可升级、定制不冲突**。详见 `brain/README.md`。
6. **部署配置有安全默认**。`loadConfig` 里每个键都 `?? 默认值`，且 `config.json` 缺失回退 `config.example.json`；生产模式由 `BAILING_ENV=production` 显式开启，敏感项必须来自环境变量或密钥管理器。

## 四、破坏性变更的标准流程（major 才会触发，这是"低入侵"的承诺所在）

真要改签名构造、删/改一个头或一个 schema 列、改嵌入属性名时，**必须**走完整流程，绝不"换个公式/改个列直接发新版"：

1. **新名不复用已有语义**：引入 `sha256-2=` / 新列名 / 新属性名 / bump `WIDGET_API`，与现存的并存、可区分。
2. **过渡窗内双发或双收**：在 ≥ 1 个完整 minor 版本里，hub 能**同时发两种格式**（或同时收两种格式），让部署方**先升 hub、再逐个升业务系统 SDK / 嵌入页**，分步完成、无停机硬切。
3. **CHANGELOG 写死时间表**：哪个版本起弃用（deprecated）、哪个版本起移除（removed），白纸黑字。
4. **永不静默改行为**：hub 见到弃用标签/字段会**记审计 + 告警**，而不是悄悄按新逻辑跑——升级出问题要能一眼看见。

## 五、DB schema 演进纪律（贡献者必读）

写数据库结构文件前先读 **`sql/README.md`**。三条铁律：

1. **永远新增编号文件，绝不编辑已发布过的 `sql` 文件**。已部署实例会按账本跳过已应用文件；改已发布文件会造成不同实例之间的 schema 分叉。
2. **发布后只准 `CREATE TABLE IF NOT EXISTS` 与 `ADD COLUMN ... DEFAULT ...`**（加索引同理可幂等）。这类是纯增量、drop-in。
3. **禁 `RENAME` / `DROP COLUMN` / `DROP TABLE` / 改列义**。要删要改名 → 走第四节：先加新列双写、过渡窗、下个 major 再删旧列。
   - 尤其 `RENAME`：它让结构变更只能在特定状态下执行；一旦换库或重置后重放，容易中断初始化——能不用就不用。

## 六、嵌入组件契约（`WIDGET_API`）

`widget.js` 由 hub 托管，业务页面 `<script src=".../widget.js?...">` 引的永远是 hub 当前版本——**hub 升级当晚，所有嵌入页自动换新版 widget**。所以嵌入面是和签名同级的 wire 契约，受 SemVer 约束（见 `widget.js` 头部 `WIDGET_API`）。稳定面：

- **脚本属性**：`data-entry`（必填）、`data-open`、`data-ticket`
- **全局对象**：`window.BailingChat.setContext({page_key,page_name})`、`window.BailingChat.apiVersion`
- **端点族**：`/chat/:entry`、`/chat/:entry/config`、`/chat/:entry/events/:jobId`、`/chat/:entry/thread`、`/chat/:entry/rate/:jobId`、`/chat/:entry/upload`

`/chat/:entry/events/:jobId` 是聊天组件主结果流端点，用于接收 `open/status/ping/done/failed/timeout` 事件。断线恢复和跨设备打开统一通过 `/chat/:entry/thread` 回灌服务端会话总账。

加属性/加可选端点 = `WIDGET_API` 不变、既有页面照跑；改属性名/改端点语义 = bump `WIDGET_API` + 过渡窗，绝不静默改。

> **会话是一等公民，历史 UX 由你定。** 每段对话在中枢都是可寻址的 thread（组件携带 `thread_id` 分区，「开启新对话」=换一个），全量留存在对话总账。中枢只给原语、不强加形态：默认气泡只提供「开启新对话」，**不内置历史会话列表**——因为客服问答气泡多半不需要，而真正想要历史的登录态工作台，通常更愿意用自己的设计在自己的 app 外壳里呈现。需要历史列表时，可基于 `/chat/:entry/thread`（按身份+`thread_id` 拉某条线程总账）的模式自行扩展你那侧的 UI；如需「列出某身份的全部会话」，这是一层薄读视图（数据已在总账），按需提 issue 即可加为可选端点。

## 七、部署方升级清单（贴在运维手册里）

1. `git pull` / 拉新 tag → 看 [CHANGELOG.md](CHANGELOG.md)。
2. **没有 `BREAKING` 段** → `db:init` + 重启 hub 即可（patch/minor，drop-in），路由/凭证/入口零重配。
3. **有 `BREAKING` 段** → 按其指引：先升 hub（过渡窗内双发/双收），再依次把各业务系统的 SDK、嵌入页升到新版；用 `docs/examples/bailing-tool-verify.{php,mjs,py}` 跑冻结测试向量自检对齐后再灰度。
4. 任何时候 hub 日志出现"弃用标签/字段"告警 = 该项在下个 major 会移除，安排升级即可，不影响当前运行。

---

**一句话**：加东西走 minor、放下就走；改/删（wire / schema / 嵌入属性）走 major、新名 + 过渡窗、永不硬切、永不静默。运行时配置在 DB、生产密钥在环境变量或密钥管理器、定制在 `.local`、键有默认——代码升级碰不到它们。下游每次跟版的"入侵成本"因此是**可预期**的，而不是每次都得赌。

## 八、每次改动必须同步的开发者说明

公开发布后按同一条纪律执行：凡是影响业务侧对接、wire 字段、SDK、数据库结构、控制台配置路径的改动，合并前必须同步三处：

1. **CHANGELOG**：写清楚类型、影响面、业务侧是否要改、怎么验证；
2. **CONTRACT / 对接指南**：如果字段或调用方式变了，更新稳定契约和开发者操作手册；
3. **控制台说明**：如果后台路径或配置项变了，更新页面内接入说明或示例代码。

部署后可用 `GET /version` 查看当前代码侧契约版本；后台「系统状态」页可查看数据库结构是否已经应用到当前版本。
