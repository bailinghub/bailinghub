# Runbook：工单初查

适用：业务侧提交的一般性工单 / bug 报告。目标是在只读范围内快速定位「是什么、可能在哪、多严重、下一步」。

## 步骤

1. **读懂诉求**：把工单正文转成一句话问题陈述。信息不足（无报错、无复现、无时间/租户）→ `needs_human=true`、`category=需更多信息`，列出还缺什么，结束。
2. **定位现场**：
   - 有报错信息 → 全库 grep 报错关键字 / 异常类名 / 错误码。
   - 涉及某功能 → 从路由/控制器入口找到对应 Service/Model。
   - 涉及数据 → 用只读 DB 工具核对相关表的真实状态（只 SELECT）。
3. **看近期变更**：`git log` / `git show` 最近改动相关文件的提交，判断是否回归。
4. **看运行证据**：相关日志（参考项目 memory 里记录的日志位置/查询方式）。注意 OPcache、daemon 缓冲、reload/restart 等已知陷阱。
5. **形成结论**：根因假设 + 证据（落到 `文件:行号`/日志/表）+ 严重度。
   - 影响交易/支付/核销/资金 → P0/P1。
   - 单点功能异常、有 workaround → P2。
   - 体验/文案/边缘 → P3。
6. **给建议**：`suggested_next_step`；若能想到修复方向，填 `proposed_actions`（标注 risk/可逆），但**不执行**。

## 红线

- 全程只读。任何写/部署/重启都不做，只能作为 `proposed_actions` 建议项。
- 不复述、不外泄任何密钥/凭据明文。
- 宁可 `needs_human=true`，不要自信的错误根因。
